# ADR-001 
ADR-010: 클린 아키텍처 기반 계층 구조 및 의존성 규칙 채택

## 1. Context
- 본 프로젝트는 콘서트 예매·대기열·좌석 배정 등 복잡한 도메인 모델을 다루는 서비스로,
기능 확장에 따라 도메인 복잡도가 증가할 것이 예상된다.

- 대기열 및 판매 시작 시점 등의 순간적인 트래픽 증가로 인한 부하 발생 상황을 고려하고, 
이러한 상황을 가정한 수평 확장(Scale-out) 가능한 구조 설계를 목표로한다. 

- 비동기 이벤트 처리(Kafka, SQS),
향후 CQRS 도입 가능성 등 다양한 기술 요구사항을 고려한다.

- 도메인 로직을 인프라스트럭처 변화와 분리하여 유지보수성을 높이기 위해,
설계 초기 단계부터 패키지 구조와 의존성 방향을 명확히 규정할 필요가 있다.
## 2. Decision
- 이에 따라 Clean Architecture 스타일을 채택하고
   Domain → Application → web → Infra 계층 구조를 따르기로 한다.

## 3. Rationale
- 핵심 비즈니스 규칙을 외부 변화로부터 보호
- 테스트 용이성 증가
- 도메인 모델을 명확히 정의
- 파일 업로드, 이벤트 처리, SQS/Lambda 같은 인프라 교체가 용이
- UseCase 기반으로 애플리케이션 흐름을 명확히 컨트롤함

## 4. Alternatives
- 전통 계층형 아키텍처(Layered Architecture) → Controller-Service-Repository
- 도메인 모델이 빈약해지고 서비스에 모든 로직이 집중됨 → 폐기
- 헥사고날 아키텍처 → 개념적으로 유사하지만 표준화된 문서가 약간 모호
- 모놀리틱 중심 서비스 → 장기 유지보수 관점에서 불리

## 5. Consequences

좋은 점:
- 유스케이스 중심의 구조로 유지보수성 향상
- 도메인 규칙 일관성 확보
- 테스트 용이성 증가
부정적 영향:
- 초기 설계 복잡도 증가
- 러닝 커브
